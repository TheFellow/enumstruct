# enumstruct

A `go/analysis` linter that enforces exhaustive nil-check switches on **pointer-union structs** — structs where every field is a pointer type and exactly one is expected to be non-nil at runtime.

This pattern commonly arises from GraphQL code generation (e.g. `gqlgen`) where a union input type becomes a Go struct with all-pointer fields.
Without a linter, adding a new variant silently drops it through to the default/fallthrough case.

## The problem

GraphQL doesn't have a union input type, so tools like `gqlgen` represent a "one of these" input as a struct with all-pointer fields:

```go
// Generated by gqlgen — adding a new payment method here...
type PaymentMethodInput struct {
    Card         *CardInput
    BankTransfer *BankTransferInput
    Crypto       *CryptoInput
}
```

Code that consumes it uses a nil-check switch:

```go
// ...must also be updated here, or the new variant silently falls through.
switch {
case input.Card != nil:
    return processCard(input.Card)
case input.BankTransfer != nil:
    return processBankTransfer(input.BankTransfer)
// input.Crypto added to the struct but missing here — no compiler error
}
```

Similar scenarios can occur when trying to serialize a Go sum type to/from JSON.

`enumstruct` catches the missing case at lint time.

## Installation

```bash
go install github.com/TheFellow/enumstruct/cmd/enumstruct@latest
```

## Usage

```bash
enumstruct ./...
```

As a `go/analysis` analyzer, it also integrates into any analysis-compatible harness (golangci-lint custom analyzers, `go vet` plugin pipelines).

## Declaring pointer-union structs

### Annotation (for code you own)

Place `//enumstruct:decl` immediately above the struct declaration:

```go
//enumstruct:decl
type NotificationEvent struct {
    Email *EmailEvent
    Sms   *SmsEvent
    Push  *PushEvent
}
```

### Config file (for generated or imported code)

Create `.enumstruct.yml` in your module root (or any ancestor directory):

```yaml
types:
  - "github.com/org/api/model.PaymentMethodInput"
  - "github.com/org/proto/gen.NotificationEvent"

default_mode: strict       # strict | lenient  (default: strict)
check_generated: true      # lint generated files (default: true)

exclude_fields:
  "github.com/org/api/model.PaymentMethodInput":
    - LegacyWireTransfer
```

Types are always specified as `<full_import_path>.<TypeName>`. The linter resolves them via `go/types` — no filesystem path assumptions.

## Directives

| Directive                             | Placement                | Effect                                      |
|---------------------------------------|--------------------------|---------------------------------------------|
| `//enumstruct:decl`                   | Above a struct type      | Marks the struct as a pointer-union         |
| `//enumstruct:ignore`                 | Above a switch statement | Suppresses exhaustiveness for that switch   |
| `//enumstruct:ignore-field FieldName` | Above a struct type      | Excludes a field from exhaustiveness checks |

## What gets checked

### Missing cases

```go
//enumstruct:decl
type Union struct {
    A *TypeA
    B *TypeB
    C *TypeC
}

func handle(u Union) {
    switch {          // error: missing cases: C
    case u.A != nil:
        // ...
    case u.B != nil:
        // ...
    }
}
```

### Duplicate cases

```go
switch {
case u.A != nil:
case u.A != nil:  // error: duplicate nil-check case for field A
case u.B != nil:
}
```

### Non-pointer fields on annotated structs

```go
//enumstruct:decl
type Bad struct {           // error: field "Metadata" of Bad is not a pointer type
    A        *TypeA
    B        *TypeB
    Metadata string         // must be a pointer or excluded with //enumstruct:ignore-field
}
```

## Suppression

Suppress a specific switch that intentionally handles a subset of variants:

```go
//enumstruct:ignore
switch {
case u.A != nil:
    // only handling A here on purpose
}
```

The directive tolerates up to two blank lines between it and the `switch` keyword.

Exclude a deprecated or metadata field globally:

```go
//enumstruct:decl
//enumstruct:ignore-field LegacyVariant
type Union struct {
    A             *TypeA
    B             *TypeB
    LegacyVariant *LegacyType  // not required in switches
}
```

## Modes

**Strict (default):** A `default:` clause does not satisfy exhaustiveness — every field must have an explicit `case`. This is the correct choice when `default: panic("unreachable")` indicates the author intended full coverage.

**Lenient:** A `default:` clause exempts the switch from exhaustiveness checks. Set globally via `.enumstruct.yml`:

```yaml
default_mode: lenient
```

## Cross-package support

When a struct is annotated with `//enumstruct:decl`, the linter exports an `analysis.Fact` so the exhaustiveness check is enforced in any package that consumes the type — not just the package that defines it.

For config-registered types, the linter resolves them via `pass.Pkg.Imports()` (direct imports only).

## Pattern recognition

The case-expression recognizer handles all real-world nil-check forms:

| Form                                                  | Recognized          |
|-------------------------------------------------------|---------------------|
| `u.Field != nil`                                      | Yes                 |
| `nil != u.Field`                                      | Yes (reversed)      |
| `(u.Field != nil)`                                    | Yes (parenthesized) |
| Pointer receiver: `(*Union).Field != nil`             | Yes                 |
| Multi-expression case: `case u.A != nil, u.B != nil:` | Yes                 |

Matching is done via `*types.Var` identity from `pass.TypesInfo.Selections`, never by field name strings. This prevents false matches from promoted or embedded fields and ensures correctness across packages.

## Known limitations

- `if/else if` chains are not checked (v1 scope: `switch {}` only)
- Config types must be direct imports of the package being analyzed (transitive imports are not resolved)
- `//enumstruct:decl` on a spec inside a grouped `type (...)` block requires the annotation to be directly above the type name, not above `type (`
